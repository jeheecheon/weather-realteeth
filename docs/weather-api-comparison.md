# 일기 API 비교 및 선정

## 결론: Open-Meteo 선택

비교군(Open-Meteo / OpenWeatherMap / 기상청) 중 **Open-Meteo**를 사용한다.

이 앱은 SSR이나 SEO가 필요한 지점이 없다.
기본적으로 단일페이지 프로젝트이며, 브라우저 위치 정보(`navigator API`)에 의존해 서버 렌더링을 하기에는 다소 무리가 있는 구조였다. 게다가 SEO가 요구사항이 아니며, 페이지 특성 자체로 검색 노출에 의미가 없다. 따라서 별도 백엔드가 없는 순수 SPA가 가장 잘 맞고, Open-Meteo라는 API는 이 구조에 정확히 들어맞는다.

Open-Meteo 선택 근거를 정리하면 아래와 같다.:

- 키 발급이 필요가 없다.
- 사용량 한도가 IP 단위라 사용자가 늘어도 자연스럽게 분산된다.
- CORS가 열려 있어 브라우저에서 직접 호출할 수 있고, 무엇보다 프록시 서버를 둘 필요가 없다.
- 최소 요구사항에 맞는 데이터를 모두 제공한다.

다만 한국 국지 예보(특히 강수) 정확도는 기상청보다 낮고, 날씨 아이콘은 WMO 코드를 직접 매핑해야 한다. 아이콘 매핑은 한 번 만들어 두면 끝나는 작업이라 감수할 만하다.

## 비교군

PRD 요구사항(현재 기온 / 당일 최저/최고 / 시간대별 기온 / 위치 감지 / 장소 검색) 기준으로 세 비교군을 비교했다.

| 항목                 | Open-Meteo         | OpenWeatherMap       | 기상청 (KMA)            |
| -------------------- | ------------------ | -------------------- | ----------------------- |
| API 키 / 가입        | 불필요             | 필요 (무료)          | 필요 (무료)             |
| 좌표계               | 위경도             | 위경도               | 격자(nx, ny) 변환 필요  |
| 당일 최저/최고       | 응답에서 직접 제공 | forecast로 직접 계산 | 직접 제공 (TMN/TMX)     |
| 시간대별 기온        | 1시간 간격         | 3시간 간격           | 1시간 간격              |
| 날씨 아이콘          | WMO 코드 직접 매핑 | 아이콘 URL 제공      | 코드 직접 매핑          |
| 한국 정확도          | 보통 (글로벌 모델) | 보통 (글로벌 모델)   | 최고 (한반도 전용 모델) |
| 클라이언트 직접 호출 | 가능 (CORS 허용)   | 가능 (CORS 허용)     | 불가 → 프록시 필요      |
| 구현 난이도          | 낮음               | 중간                 | 높음                    |

## 사용량 한도

단순 호출 수보다 한도가 누구 기준으로 적용되는지가 실제 병목을 결정한다. 기상청은 프록시를 거쳐야 해서 모든 사용자 트래픽이 인증키 하나에 합산되는 반면, Open-Meteo는 사용자 IP별로 한도가 적용된다.

|                | 한도                          | 적용 단위                   |
| -------------- | ----------------------------- | --------------------------- |
| Open-Meteo     | 일 약 10,000                  | IP당 (사용자별 분산)        |
| OpenWeatherMap | 일 1,000,000                  | API 키당                    |
| 기상청         | 일 20,000 (API 허브 일반회원) | API 키당 (전체 사용자 합산) |
